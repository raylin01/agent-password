# AgentPass V1.1 Build Plan

Date: April 9, 2026
Status: Locked Plan For Next Implementation Pass

## Purpose

This plan defines the V1.1 changes to AgentPass after V1.

The goal is to add:

- a free-form secret entry type
- hidden internal keys
- automatic label deduplication

This document is intentionally implementation-ready. The next pass should be able to code directly against it.

## Summary Of Changes

V1.1 adds a third entry type:

- `secret`

Unlike `login` and `card`, a `secret` entry supports arbitrary user-defined fields.

V1.1 also changes the `key` model:

- users no longer create or edit `key`
- `key` becomes an internal slug only
- `key` is generated from the final deduped label
- labels are auto-deduped at creation time

Examples:

- `Costco`
- `Costco (1)`
- `Costco (2)`

Internal keys:

- `costco`
- `costco-1`
- `costco-2`

## Locked Product Decisions

### 1. Free-Form Secret Fields

The new `secret` entry type uses arbitrary named fields.

Each field has:

- `name`
- `value`
- `handle`
- `policy`
- timestamps

There is no extra `kind` field in V1.1.

Reason:

- `kind` adds complexity without enough value right now
- file replacement and browser fill already work from handles
- users can express meaning through field names like `api_token`, `zone_id`, `client_secret`, `DATABASE_URL`, or `region`

### 2. Hidden Key Model

`key` becomes internal-only.

Rules:

- UI must not ask for `key`
- CLI create commands must not require `key`
- API create endpoints must not require `key`
- system generates `key` from deduped label

### 3. Label Deduplication

Creation-time dedupe rule:

- if `Costco` does not exist, create `Costco`
- if `Costco` exists, create `Costco (1)`
- if `Costco` and `Costco (1)` exist, create `Costco (2)`

Normalization rule:

- compare labels case-insensitively for dedupe
- trim surrounding whitespace before comparison

We will not allow duplicate labels after creation.

### 4. Rename Behavior

When a user renames an entry:

- the label is deduped again if needed
- the internal `key` is regenerated from the final label
- handles are regenerated from the new key

Reason:

- hiding `key` means key stability is no longer part of the user contract
- the product should keep label, key, and handles aligned

Tradeoff:

- handle names will change on rename
- this is acceptable in V1.1 because hidden keys and user-visible labels should stay consistent

This must be documented clearly.

## V1.1 Scope

V1.1 will include:

- new `secret` entry type
- arbitrary fields for secret entries
- hidden internal keys everywhere
- automatic label dedupe on create
- automatic label dedupe on rename
- key regeneration on rename
- handle regeneration on rename
- UI support for adding and editing free-form secret fields
- CLI support for adding and editing secret entries
- tests for dedupe and handle regeneration

V1.1 will not include:

- field kinds
- per-field grouping or folders
- handle aliases
- backward-compatible stable handles across rename

## Data Model Changes

### Entry Types

After V1.1, supported entry types are:

- `login`
- `card`
- `secret`

### Secret Entry Shape

`secret` entry metadata:

- `id`
- `entryType`
- `label`
- `key`
- `provider`
- `notes`
- `tags`
- `createdAt`
- `updatedAt`
- `fields`

`provider` is optional metadata only.

### Secret Field Shape

Each secret field contains:

- `id`
- `entryId`
- `fieldName`
- `fieldType`
- `handle`
- `value`
- `previewMasked`
- `policy`
- `createdAt`
- `updatedAt`
- `lastUsedAt`

For `secret` entries:

- `fieldType` should equal `fieldName`

Example:

- label: `Cloudflare`
- provider: `Cloudflare`
- field name: `api_token`
- handle: `CLOUDFLARE_API_TOKEN_1`

### Existing Typed Entries

`login` and `card` remain typed entries with fixed expected fields.

V1.1 must preserve current behavior for those entry types.

## Key And Handle Rules

### Label To Key

Key generation rule:

1. take final deduped label
2. normalize to lowercase slug
3. replace non-alphanumeric runs with `-`
4. trim leading/trailing `-`

Examples:

- `Costco` -> `costco`
- `Costco (1)` -> `costco-1`
- `AWS Prod` -> `aws-prod`

### Key To Handle

Handle generation stays:

- `NORMALIZED_KEY + FIELD_NAME + INDEX`

Examples:

- label `Cloudflare`
- key `cloudflare`
- field `api_token`
- handle `CLOUDFLARE_API_TOKEN_1`

### Rename Rule

On rename:

1. dedupe new label
2. derive new key
3. regenerate all field handles for that entry

This applies to:

- `login`
- `card`
- `secret`

## API Changes

### Create Endpoints

V1 create endpoints should be changed so `key` is not required.

`POST /api/entries/login`

- remove `key` requirement
- derive key from deduped label

`POST /api/entries/card`

- remove `key` requirement
- derive key from deduped label

New endpoint:

`POST /api/entries/secret`

Payload:

```json
{
  "label": "Cloudflare",
  "provider": "Cloudflare",
  "notes": "Production account",
  "tags": ["infra", "prod"],
  "fields": [
    { "name": "api_token", "value": "..." },
    { "name": "account_id", "value": "..." },
    { "name": "zone_id", "value": "..." }
  ]
}
```

### Update Endpoints

`PATCH /api/entries/:entryId`

Must support:

- label rename with dedupe
- provider update for `secret`
- notes update
- tags update

Rename side effect:

- key regeneration
- handle regeneration

### Field Endpoints

Existing field update and remove endpoints stay.

New or updated add-field behavior:

- `POST /api/entries/:entryId/fields`
- must support `secret` entries with arbitrary field names
- must continue supporting missing fixed fields on `login` and `card`

## CLI Changes

### New Command

Add:

- `agentpass add-secret`

Example:

```bash
agentpass add-secret \
  --label "Cloudflare" \
  --provider "Cloudflare" \
  --field api_token=abc123 \
  --field account_id=acct_123 \
  --field zone_id=zone_456
```

### CLI Rules

- remove `key` from `add-login`
- remove `key` from `add-card`
- do not show internal key in standard create UX
- keep showing key in `list` output for now only if useful for debugging

Recommended output change:

- prefer showing `label` and handles
- `key` may remain in JSON output but should be omitted from the normal human-friendly output

### Prompting

`add-secret` should support repeated `--field name=value`.

Optional prompt mode:

```bash
agentpass add-secret --label "AWS Prod" --provider AWS --prompt
```

Prompt flow:

1. ask for field name
2. ask for field value
3. repeat until blank field name

## UI Changes

### Create Forms

Change login form:

- remove visible `key` input

Change card form:

- remove visible `key` input

Add secret form:

- `label`
- `provider`
- `notes`
- `tags`
- repeating field rows:
  - `field name`
  - `field value`

### Entry Cards

For `secret` entries:

- show provider if present
- show free-form fields like any other entry
- allow adding new custom fields from the UI

### Entry Editing

Rename should:

- update label
- trigger dedupe if needed
- regenerate key and handles

UI should show new handles after rename.

### UX Requirement

The user must never need to think about internal keys.

## Service Layer Changes

### New Helpers

Add helper:

- `dedupeLabel(existingEntries, requestedLabel, excludingEntryId?)`

Behavior:

- compares labels case-insensitively
- returns deduped label

Add helper:

- `deriveEntryKeyFromLabel(label)`

Add helper:

- `regenerateEntryHandles(entry)`

### Create Entry Logic

For all entry types:

1. require label
2. dedupe label
3. derive key from label
4. create entry

For `secret`:

5. create arbitrary fields from provided field list

### Rename Logic

For all entry types:

1. dedupe new label
2. derive new key
3. update entry label and key
4. regenerate field handles
5. save and audit

## Audit Logging Changes

Audit actions to add:

- `entry.secret.add`
- `entry.label.deduped`
- `entry.rename`
- `field.handle.regenerated`

Minimum logging rule:

- whenever label dedupe changes the user-requested label, log both requested and final label in a redacted metadata field

No raw secret values should ever appear in logs.

## Migration Plan

### Existing Data

Existing V1 entries already have keys.

Migration strategy:

- keep current stored keys and labels during migration
- do not regenerate existing keys on migration
- only use new hidden-key logic for:
  - new entries
  - renamed entries after V1.1

Reason:

- safer migration
- avoids surprise handle churn on upgrade

### Schema Version

Bump vault schema version from `2` to `3`.

Migration from V2 to V3:

- preserve existing entries
- add optional `provider` field where absent
- mark schema version `3`

## Acceptance Criteria

V1.1 is complete when all are true:

- users can create secret entries with arbitrary named fields
- users do not need to provide a key anywhere
- labels are auto-deduped on create
- labels are auto-deduped on rename
- keys are generated internally from labels
- handles are generated from keys plus field names
- handles regenerate on rename
- login and card flows still work without visible keys
- CLI supports `add-secret`
- UI supports creating and editing secret entries
- file rendering works with secret-entry fields
- browser fill works with secret-entry fields
- tests cover dedupe, rename, and free-form fields
